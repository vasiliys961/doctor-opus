# Doctor Opus — техническая и клиническая презентация платформы

> Версия документа: июль 2026. Подготовлено для демонстрации специалистам по ИИ и врачам-экспертам, в т.ч. как материал к предложению о клиническом испытании.

**Как читать документ:** по каждому разделу приложения — два блока:

- 🔧 **Технический блок** (для ИИ-специалиста) — стек, модели, промпты, архитектура.
- 🩺 **Медицинский блок** (для врача) — что это даёт клинически, глубина анализа, как использовать.

---

## Executive Summary

Doctor Opus — веб-платформа поддержки врачебных решений (CDSS) на Next.js 14, объединяющая мультимодельный ИИ-анализ (через единый шлюз OpenRouter) с прикладной медицинской логикой: от вольтажных критериев ЭКГ и ACR/BI-RADS-протоколов до генетической интерпретации по ACMG и мультиагентного консилиума в стиле Microsoft MAI-DxO. Все данные проходят обязательную анонимизацию (текст — всегда, изображения/DICOM — опционально по флагу), биллинг ведётся пофайлово в PostgreSQL с блокировками `FOR UPDATE`, инфраструктура изолирована по контурам RU/EN.

Ключевая архитектурная идея — **двухэтапный workflow** («Vision → структурированный JSON → клиническое рассуждение») почти во всех модулях: дешёвая модель извлекает находки в машиночитаемом виде, дорогая модель строит на этом клиническую директиву. Это снижает стоимость и повышает воспроизводимость по сравнению с одним end-to-end вызовом.

## 0. Общая архитектура платформы

### 🔧 Технический блок

- **Стек:** Next.js 14 (App Router, TypeScript), PostgreSQL (Neon/self-hosted), NextAuth (JWT-стратегия), Docker Compose + Nginx на Timeweb VPS.
- **Единый AI-шлюз — OpenRouter.** Модели не хардкожены под конкретного провайдера — при отказе Anthropic автоматически идёт fallback на GPT-5.4 или Gemini.
- **Точный реестр моделей (`lib/openrouter.ts` → `MODELS`), актуально на июль 2026:**

| Роль в системе | OpenRouter ID | Где используется |
|---|---|---|
| Базовый Opus (режим optimized/direct) | `anthropic/claude-opus-4.6` | Точный анализ изображений по умолчанию, сравнительный анализ, чат |
| Opus для режима **validated** | `anthropic/claude-opus-4.8` (откат на `4.7` через env `VALIDATED_OPUS_MODEL`, без деплоя) | Максимальная точность в `/image-analysis`, `/xray`, `/ct`, `/mri` и др. при выборе режима "validated" |
| Sonnet | `anthropic/claude-sonnet-5` | Этап 2 обычного анализа, роли Test-Chooser/Stewardship и мнения специальностей в Консилиуме |
| GPT | `openai/gpt-5.4` | Geo/permission fallback для Anthropic-моделей |
| Быстрая vision-модель | `google/gemini-3-flash-preview` | Vision JSON (этап 1), OCR, triage, маршрутизация специальностей в Консилиуме |
| Pro vision-модель | `google/gemini-3.1-pro-preview` | Эскалация при сложных/неоднозначных снимках (КТ, МРТ, гистология, маммография) |
| **Fable 5** | `anthropic/claude-fable-5` | **Используется именно и только в Консилиуме** — роли Dr. Hypothesis, Dr. Challenger, Dr. Checklist (клиническая глубина рассуждения в дебатах) |
| Резерв для Fable | `sakana/fugu-ultra` | Автоматический технический fallback, если Fable 5 недоступна |

> **Важно по Fable 5:** это не модель "по умолчанию для всех" — она подключена целенаправленно только к трём ролям дебатов Консилиума (Hypothesis/Challenger/Checklist), где по данным HealthBench Professional показывает более высокий результат, чем Opus и GPT-5.5. Остальные роли Консилиума (Test-Chooser, Stewardship, синтез AMSC, маршрутизация) работают на Sonnet 5 — это сознательный баланс стоимости и клинической глубины, а не ограничение возможностей.
- **Двухэтапный pipeline для изображений:**
  1. Этап 1 (Vision JSON) — быстрая модель (обычно Gemini 3 Flash) извлекает находки в строгий JSON: `findings`, `metrics`, `is_critical`, `confidence`.
  2. Этап 2 (клиническое рассуждение) — более сильная модель (Sonnet 5 / Opus 4.8 / GPT-5.4) строит на этом JSON полноценную клиническую директиву по единому промпту `STRATEGIC_SYSTEM_PROMPT`.
- **Smart routing:** при `self_confidence < 0.75`, высокой заявленной сложности или high-risk модальности (КТ, МРТ, гистология, маммография) происходит эскалация с Gemini Flash на Gemini 3.1 Pro без участия пользователя.
- **Streaming:** SSE через `TransformStream` с padding (2–4 КБ) против буферизации прокси, heartbeat каждые 5 с, агрегация usage с обоих этапов для точного биллинга.
- **Адаптивные лимиты токенов** (`lib/adaptive-tokens.ts`) — `max_tokens` рассчитывается по длине диалога/файлов/режима, а не берётся фиксированным.
- **Единый промпт-модуль** `lib/prompts.ts` (~1900 строк) — все системные промпты и критерии по модальностям живут в одном месте, что упрощает аудит и версионирование (`SPECIALTY_PROMPT_VERSIONS`).

### 🩺 Медицинский блок

- Врач получает не «чёрный ящик», а прозрачную двухступенчатую логику: сначала система формально фиксирует, *что она видит* на снимке (числа, находки), и только потом строит из этого клиническое заключение — это снижает риск того, что модель «додумает» находки, которых нет.
- Автоматическая эскалация на более мощную модель при подозрении на сложный/неоднозначный случай происходит без участия врача — не нужно самому решать, какую модель выбрать для сложного КТ.
- Потоковый вывод ответа (посимвольно) даёт врачу возможность читать рассуждение по мере генерации, а не ждать полного ответа.

## 1. ЭКГ

### 🔧 Технический блок

- Критерии зашиты в `SPECIALIST_CRITERIA.ecg` (`lib/prompts.ts`): извлекаются вольтаж, скорость записи, ЭОС, интервалы (P, PQ, QRS, QT/QTc), морфология ST, зубцы, признаки гипертрофии и блокад.
- Этап 1 (Gemini Flash) помечает критические находки флагом `is_critical=true` (STEMI, ФЖ, AV-блокада III степени, QTc > 500).
- Этап 2 использует `STRATEGIC_SYSTEM_PROMPT`, который жёстко требует выносить блок «КРИТИЧЕСКОЕ СОСТОЯНИЕ» первым абзацем отчёта, если он есть.
- Отдельный инструмент — `EcgCaliper.tsx`, ручной калипер измерения интервалов по пикселям с калибровкой по скорости (25/50 мм/с) — работает независимо от ИИ, как assistive-инструмент для врача.
- Диагноз-ориентированные источники (LITFL, ECGWaves) подбираются автоматически по найденным паттернам через `lib/ecg-reference-links.ts`.
- Поддержан прямой ввод с USB-кардиографа (1/3/4/6/12 каналов) с автодетектом скорости/формата (раздел «Лаборатория»).

### 🩺 Медицинский блок

- Система распознаёт весь спектр стандартных ЭКГ-паттернов: ОКС и локализация ишемии по отведениям, ФП/ТП/экстрасистолия, WPW, синдром Бругада, признаки ТЭЛА (S1Q3T3), электролитные нарушения, интоксикация дигоксином, перикардит.
- Критические находки выносятся отдельным блоком в начале заключения — врач сразу видит «красные флаги», не читая весь текст.
- Ручной калипер полезен для верификации автоматических измерений интервалов — не приходится слепо доверять цифрам модели.
- Итоговое заключение всегда сопровождается ссылками на профильные источники (LITFL/ECGWaves) по конкретно найденным паттернам — удобно для быстрой сверки с литературой прямо в моменте.

## 2. Рентген / КТ / МРТ / УЗИ

### 🔧 Технический блок

- Каждая модальность имеет свой блок в `SPECIALIST_CRITERIA` (`xray`, `ct`, `mri`, `ultrasound`) с точным перечнем ожидаемых числовых параметров (HU, размеры в мм, RI/PI, КТИ и т.п.).
- Для «чистого» радиологического протокола (без полной клинической директивы) применяется отдельный промпт `RADIOLOGY_PROTOCOL_PROMPT`, структурированный по стандарту **FINDINGS → IMPRESSION**, с шкалами BI-RADS/TI-RADS/LI-RADS/PI-RADS.
- Native DICOM: парсинг (`dicom-parser`), просмотр в браузере, опциональная анонимизация DICOM-тегов (Patient Name/ID/Birth Date, Institution и др.) на сервере.
- Отдельный модуль `/advanced-3d` — объёмный рендеринг папки DICOM (Cornerstone/VTK.js) в режимах Clinical/Cinematic — визуализация, не автоматический диагностический вывод.
- Диагноз-ориентированные источники — Radiopaedia и Radiology Assistant, подключаемые через site-search там, где встроенный поиск сайта нестабилен.

### 🩺 Медицинский блок

- **Рентген:** оценка проекции/качества снимка, лёгочных полей, корней, плевры, средостения (КТИ), скелета; целевые находки — пневмония, пневмоторакс, гидроторакс, отёк лёгких, туберкулёз, образования, переломы.
- **КТ:** посегментный разбор с указанием HU, окнами (лёгочное/мягкотканное/костное/сосудистое), ASPECTS для инсульта, TNM-описание образований, признаки ТЭЛА и аневризм.
- **МРТ:** разбор по последовательностям (T1/T2/FLAIR/DWI/ADC/STIR), различение истинного ограничения диффузии от T2-shine-through, оценка масс-эффекта в мм, критерии МакДональда для демиелинизации.
- **УЗИ:** допплеровские индексы (RI/PI с клиническим порогом RI > 0.7), формализованные шкалы BI-RADS/TI-RADS/LI-RADS прямо в заключении.
- Возможность 3D-реконструкции DICOM-серии для наглядной демонстрации пациенту или на консилиуме, отдельно от текстового ИИ-анализа.

## 3. Дерматоскопия и кожные изменения

### 🔧 Технический блок

- `SPECIALIST_CRITERIA.dermatoscopy` требует от Vision-этапа оценить тип паттерна, палитру цветов, критерии **ABCDE**, специфические дерматоскопические признаки (пигментная сеть, глобулы, радиальная лучистость, бело-голубая вуаль), сосудистый рисунок, а также применить формальные алгоритмы — **7-point checklist Ардженциано**, правило Мензеса, **Ugly Duckling**.
- Три под-режима на одной странице (`app/dermatoscopy/page.tsx`): дерматоскопия / анализ раны / кожные изменения — для двух последних клинический фокус задаётся пользовательским промптом, так как отдельных строгих критериев в `prompts.ts` для них нет (используется универсальный fallback).
- Визуальный поиск похожих случаев по личным атласам врача — через CLIP-эмбеддинги (см. раздел «Библиотека / RAG»).

### 🩺 Медицинский блок

- Формальная оценка риска меланомы по ABCDE и признанным дерматоскопическим алгоритмам (не «на глаз», а по чек-листу), с дифдиагнозом: меланома (инвазивная/in situ), БКК, ПлКК, диспластический невус, себорейный кератоз, гемангиома, дерматофиброма.
- Для ран — оценка стадии раневого процесса, некроза, грануляций, эпителизации, признаков инфицирования.
- Возможность визуально сравнить текущий снимок с атласом типичных случаев или собственной библиотекой изображений врача — полезно при нетипичной картине.

## 4. Лабораторные данные, спирометрия, глюкозный мониторинг (CGM)

### 🔧 Технический блок

- **Общая лаборатория** (`/lab`): `SPECIALIST_CRITERIA.lab` требует анализ динамики показателей, референсов с учётом возраста/пола, расчётных индексов (CKD-EPI, HOMA, коэффициент де Ритиса), контроль panic values (Hb < 70, K⁺ > 6.5, Na⁺ < 120, глюкоза < 2.5), детекцию системных паттернов (цитолиз, холестаз, анемия) и лабораторных артефактов (гемолиз, липемия).
- **Спирометрия:** данные с USB-прибора или ручной ввод (FVC, FEV1, PEF, FEF25–75, MVV), клиентский промпт структурирует интерпретацию по **ATS/ERS** и **GOLD**, включая индекс Тиффно и признаки воздушной ловушки. Модель — GPT-5.4 с fallback на Sonnet 5.
- **CGM (глюкозный профиль):** импорт CSV (FreeStyle Libre, Dexcom, RU-форматы), метрики считаются на клиенте: **TIR** (время в диапазоне 3.9–10 ммоль/л), **TAR/TBR**, **GMI** (расчётный HbA1c), **CV** (порог высокой вариабельности > 36%), суточный профиль ночь/день. ИИ строит заключение по TIR/гипо-гипергликемиям/вариабельности поверх готовых метрик.

### 🩺 Медицинский блок

- Лабораторный модуль не просто перечисляет цифры, а ищет клинические паттерны (типовые синдромы) и сразу подсвечивает жизнеугрожающие значения (panic values) — критично для скрининга большого потока анализов.
- Интерпретация спирометрии сразу даёт готовую степень тяжести обструкции/рестрикции по международным стандартам (GOLD для ХОБЛ), а не только сырые цифры.
- Глюкозный модуль реализует по сути мини-AGP-отчёт (как в специализированных диабетологических системах): TIR/TAR/TBR, GMI, вариабельность, паттерны (утренняя заря, постпрандиальные пики) — с готовыми рекомендациями по терапии/питанию/мониторингу.

## 5. Генетический анализ

### 🔧 Технический блок

- Двухэтапный pipeline, отдельный от анализа изображений:
  1. **Извлечение** (`app/api/analyze/genetic/route.ts`): PDF/фото → Vision OCR (Gemini Flash) в формат `ГЕН;rsID;ГЕНОТИП;КОММЕНТАРИЙ`; VCF/VCF.GZ/CSV читаются напрямую как текст (gzip распаковывается на сервере). На этом этапе клинической интерпретации не происходит намеренно — разделение снижает риск смешивания OCR-ошибок с клиническими выводами.
  2. **Консультация** (`app/api/analyze/genetic/consult/route.ts`): Sonnet 5 / Opus 4.6 / GPT-5.4 строит заключение по извлечённым вариантам.
- `SPECIALIST_CRITERIA.genetics` требует классификацию по **ACMG** (Pathogenic / Likely Pathogenic / VUS), сверку с ClinVar/gnomAD/OMIM, оценку зиготности и типа мутации (миссенс, нонсенс, frameshift, делеция/дупликация).

### 🩺 Медицинский блок

- Итоговое заключение врача-генетика включает: обзор профиля, патогенные/вероятно патогенные варианты с ACMG-классификацией, **фармакогенетику** (по CPIC/PharmGKB — важно для подбора дозировок препаратов), нутригеномику, персонализированные рекомендации и показания к семейному консультированию.
- Формат позволяет загружать как «сырые» VCF-файлы секвенирования, так и уже готовые PDF-отчёты коммерческих лабораторий — не привязано к одному формату входных данных.

## 6. Консилиум («Глубокий анализ») — мультиагентная диагностика

Это наиболее сложный модуль платформы и основной кандидат для клинического испытания как самостоятельного CDSS-инструмента.

### 🔧 Технический блок

Гибридная архитектура: **AMSC** (Adaptive Multi-Specialty Consensus, быстрый параллельный слой) + **MAI-DxO** (5 функциональных ролей, полноценные дебаты) — эскалация из первого во второй происходит только при значимом расхождении мнений, что удерживает стоимость типового кейса низкой.

**Раунд 0 (AMSC):**
| Компонент | Модель | Функция |
|---|---|---|
| Маршрутизатор специальностей | Gemini 3 Flash | Подбирает 2–3 релевантные специальности под конкретный кейс |
| Мнения специальностей | Claude Sonnet 5 | Независимый дифференциальный диагноз от каждой «школы», параллельно |
| Фиксированная роль «Скептик» | Claude Sonnet 5 | Всегда участвует, отдельно ищет red flags независимо от подобранных специальностей — защита от «слепых зон» роутинга |
| Оценка расхождения | Gemini 3 Flash | Семантическая (не строковая!) кластеризация диагнозов — «ОРВИ» и «острый ринофарингит» не считаются расхождением |
| Синтез консенсуса | Claude Sonnet 5 | Финальный ответ, если расхождение ≤ порога и нет red flag |

**Эскалация в дебаты** происходит при `disagreementScore > 0.34` или обнаружении red flag скептиком (структурированный JSON-вердикт, не regex — это устраняет ложные срабатывания на обычные слова в тексте).

**MAI-DxO (до 2 раундов дебатов), 5 ролей:**
| Роль | Модель | Функция |
|---|---|---|
| Dr. Hypothesis | **Fable 5** (fallback: Fugu Ultra) | Формирует и обновляет дифференциальный ряд |
| Dr. Test-Chooser | Sonnet 5 | Предлагает наиболее информативные обследования |
| Dr. Challenger | **Fable 5** | «Адвокат дьявола» — ищет когнитивные ошибки и пропущенные альтернативы |
| Dr. Stewardship | Sonnet 5 | Контролирует избыточность назначений и ресурсные затраты |
| Dr. Checklist | **Fable 5** | Финальная проверка и формирование итогового диагноза |

Модель **Fable 5** — модель по умолчанию для этих трёх ролей, выбрана за более высокий результат на HealthBench Professional, чем у Opus и GPT-5.5; при технических сбоях автоматически подменяется на Sakana Fugu Ultra без участия пользователя. Ручной откат на Opus (для сравнения или при необходимости) — через `CONSILIUM_DEBATE_MODEL=opus`, что подключает `anthropic/claude-opus-4.8` (`OPUS_VALIDATED`), без деплоя.

**Политика безопасности эскалации к врачу (`escalation.ts`):**
- Расхождение мнений > 0.6 → обязательная проверка врачом.
- **Тяжёлый диагноз (`severe=true`) с итоговой уверенностью < 60% → обязательная проверка врачом всегда**, даже если агенты формально сошлись во мнении — это отдельное защитное правило поверх обычного консенсуса.
- Red flag от Challenger или сбой любого из агентов также форсирует ручную проверку.

Изображения перед подачей в консилиум преобразуются в структурированное текстовое описание находок (та же логика Vision JSON, что и в модулях анализа изображений) — консилиум работает с унифицированным текстовым представлением кейса независимо от исходного формата данных.

### 🩺 Медицинский блок

- По сути — симуляция реального врачебного консилиума: несколько независимых «специальностей» высказывают мнение параллельно, плюс отдельный «скептик», который по должности обязан искать, что могло быть упущено — независимо от того, какие специальности вообще были подобраны для этого случая.
- Если мнения расходятся незначительно — врач получает быстрый и дешёвый консенсус. Если расхождение существенное или есть подозрение на red flag — система автоматически переходит на более глубокое и дорогое обсуждение с контраргументами (аналог «второго мнения» и «третьего мнения» в живом консилиуме).
- Встроенное правило безопасности: **при подозрении на жизнеугрожающее состояние с недостаточной уверенностью системы кейс всегда уходит на обязательную проверку врачом**, что снижает риск «тихого» пропуска критического диагноза даже при формальном согласии всех «агентов».
- Итог — не просто текстовый ответ, а полноценный аудит-трейл: какие специальности рассматривали кейс, какие были разногласия, эскалировался ли случай, и почему.

## 7. ИИ-Ассистент (чат) и дополнительные проверочные механизмы

### 🔧 Технический блок

- Текстовый чат с выбором специальности/модели, поддержкой файлов, RAG по личной библиотеке врача, голосовым вводом (локальный Whisper).
- **RAG-gate:** при ответах с опорой на загруженные документы модель получает явную инструкцию маркировать утверждения, не подтверждённые источником — снижает риск «тихих» галлюцинаций при работе с клиническими рекомендациями.
- **Chain-of-Verification (по требованию, не встроено в каждый запрос):** кнопка «Проверить ответ» инициирует отдельный дешёвый вызов Sonnet 5 с ролью «адвоката дьявола» (`SELF_CHECK_SYSTEM_PROMPT`) поверх уже готового ответа — ищет пропущенные альтернативы и необоснованные заявления. Доступно и в чате, и в разделе разбора изображений.
- **Triage-классификатор сложности:** лёгкий вызов Gemini 3 Flash классифицирует первый вопрос диалога как `simple`/`complex` по чётким критериям (мультисистемность, red flags, редкая патология) и предлагает переключиться на Fable 5 или Консилиум — это только совет, ничего не переключается автоматически.
- Диагноз-ориентированные источники подбираются кросс-модально: система извлекает клинический фокус ответа (нормализуя термины RU/EN и ICD-коды) и подбирает релевантные ресурсы из объединённого словаря (DermNet, LITFL, ECGWaves, Radiopaedia, Radiology Assistant, POCUS Atlas, EyeWiki, AAO, ACR/BI-RADS, PathologyOutlines).

### 🩺 Медицинский блок

- Врач получает не «слепое доверие» к ответу ИИ, а два независимых инструмента контроля качества по требованию: пометку недостаточно обоснованных утверждений (RAG-gate) и полноценную самопроверку ответа второй, независимой «критикующей» моделью (Chain-of-Verification) — оба доступны одним кликом, без обязательного удорожания каждого обычного запроса.
- Triage-подсказка работает как «второй пилот»: если врач вводит сложный многосистемный случай в обычном (быстром) режиме, система сама предложит перейти на более вдумчивый уровень анализа — решение всегда остаётся за врачом.

## 8. Протоколы приёма и голосовые технологии

### 🔧 Технический блок

- **`/protocol`** — формирование протокола осмотра из записи разговора врач-пациент, голосовой диктовки объективных данных или ручного ввода; шаблоны по специальностям (`UNIVERSAL_SPECIALIST_TEMPLATES` в `lib/prompts.ts`); экспорт в **Word (.docx)**.
- **STT (распознавание речи):** локальный **Whisper** (`faster-whisper`, FastAPI-микросервис на порту 8000, GPU/CPU-инференс без внешнего API) — записи не покидают инфраструктуру для распознавания. Есть также резервный внешний STT (AssemblyAI) для облачного контура.
- **`/secure-protocol`** — экспериментальный режим с усиленным акцентом на локальность: запись → локальный STT → редактируемый черновик, без промежуточного хранения аудио на диске.

### 🩺 Медицинский блок

- Врач может параллельно с приёмом вести аудиозапись беседы и вручную вносить объективные данные (осмотр, витальные показатели) — оба источника объединяются в единый структурированный протокол.
- Использование локальной модели распознавания речи (Whisper) вместо облачного API — важный аргумент с точки зрения конфиденциальности разговора врач-пациент, особенно значимый для клинических испытаний и соответствия ФЗ-152.
- Итоговый протокол сразу экспортируется в привычный формат Word для истории болезни.

## 9. Персональная библиотека и RAG (текстовый + визуальный)

### 🔧 Технический блок

- **Текстовый RAG:** модель эмбеддингов `Xenova/multilingual-e5-small` (384 измерения, транслируется в браузере через transformers.js/WASM), документы режутся на чанки, векторизуются и ищутся семантически при ответах в чате/анализе.
- **Визуальный RAG (поиск по атласам):** модель `Xenova/clip-vit-base-patch32`. PDF-атлас рендерится постранично в canvas → каждая страница кодируется CLIP-энкодером изображений в вектор → L2-нормализация → квантование в Int8 → хранение в IndexedDB браузера. Поиск — косинусное сходство (скалярное произведение нормализованных векторов), запрос может быть текстом, загруженным фото или текущим анализируемым снимком (все три — в одном CLIP-пространстве).
- **Принципиально важно:** вся векторизация и поиск атласов происходят **локально в браузере**, данные атласа не передаются на сервер.

### 🩺 Медицинский блок

- Врач может загрузить собственную библиотеку клинических руководств, атласов дерматоскопии/рентгенологии и т.п., и система будет автоматически подтягивать релевантные выдержки из них при ответах в чате или анализе снимков.
- Визуальный поиск по атласу («найди похожий случай») работает как для текстового описания находки, так и для фото — полезно при нетипичной клинической картине, когда нужно сверить с собственным архивом типичных случаев, а не только с общими источниками в интернете.
- Локальная обработка библиотеки — данные личного архива врача не покидают его браузер.

## 12. Безопасность и анонимизация данных (ФЗ-152)

### 🔧 Технический блок

Четыре уровня защиты (в README заявлены как три + текстовая, фактически — четыре независимых механизма):

1. **Серверная (изображения + DICOM):** Node.js `canvas`, закрашивание краёв (верх/низ/бока) чёрными прямоугольниками; для DICOM — дополнительно затирание текстовых тегов (Patient Name/ID/Birth Date, Institution, Referring Physician и др.). Управляется флагом `maskImage`, который **включён по умолчанию** (`true`) на сервере во всех соответствующих API (`/api/analyze/image`, `/api/analyze/lab`, `/api/analyze/lab-images`, `/api/scan/document*`) — если клиент вообще не передал флаг, сервер считает его включённым. На 7 из 10 страниц анализа изображений врач может осознанно отключить маскирование отдельным чекбоксом (например, если снимок уже обрезан вручную или зона интереса — на самом краю кадра, как при дерматоскопии крупным планом). Это независимо от флага «не сохранять результат в карту пациента» — до 04.07.2026 они были ошибочно объединены в один переключатель с дефолтом "выключено", что и было точкой риска; исправлено.
2. **Быстрая клиентская:** тот же принцип закрашивания краёв, но в браузере (Canvas API) до отправки на сервер, режимы `strict`/`soft`.
3. **Точная ручная:** редактор с кистью (`ImageEditor.tsx`), undo до 10 шагов — для нестандартного расположения персональных данных (например, в центре снимка).
4. **Текстовая (`lib/anonymization.ts`):** каскад регулярных выражений с медицинским словарём исключений — маскирует ФИО, даты, паспорт, ИНН, СНИЛС, телефоны, email, адреса. Применяется **всегда**, ко всем текстовым промптам и контексту, во всех значимых API (включая генетический модуль, где до 04.07.2026 была условная логика — исправлено).

Отдельный контур для **видео** — покадровое извлечение с анонимизацией каждого кадра (безопасный режим по умолчанию) либо загрузка полного видео с обязательным явным подтверждением врача (расширенный режим).

**Безопасность инфраструктуры:** Middleware с обязательной JWT-проверкой (NextAuth) на все API кроме явного whitelist; серверный биллинг с блокировками строк `FOR UPDATE` (защита от race conditions при списании средств); автоматическая маскировка API-ключей/токенов/email в логах; API-ключи провайдеров никогда не передаются на клиент.

### 🩺 Медицинский блок

- Многоуровневая система даёт врачу выбор степени контроля: от автоматической защиты до точечной ручной анонимизации нестандартных случаев — что важно при работе с реальными снимками пациентов в рамках клинического испытания.
- Текстовая анонимизация работает без исключений и на любом текстовом входе (протокол, чат, лабораторные данные), что закрывает основной канал утечки персональных данных даже если врач забыл включить защиту изображения.
- Рекомендация для протокола клинического испытания: явно включать чекбокс серверной анонимизации изображений на всех этапах загрузки данных пациентов, так как по умолчанию он не активен.

## 13. Биллинг и учёт стоимости

### 🔧 Технический блок

- Единица измерения — «у.е.» (≈ $0.01), стоимость каждого вызова считается по фактическому числу токенов (`lib/cost-calculator.ts`) отдельно для входа/выхода по прайсу конкретной модели.
- Двухфазное списание: предварительная оценка стоимости до старта анализа → фактическое досписание/возврат после получения реального usage от OpenRouter (реконсиляция).
- Хранение — PostgreSQL (`user_balances`, `credit_transactions`, `guest_balances`), транзакционная защита `FOR UPDATE`.

### 🩺 Медицинский блок

- Прозрачная и предсказуемая стоимость каждого клинического запроса, важно для расчёта бюджета клинического испытания на конкретное число кейсов/визитов.

## Заключение: почему это релевантно для клинического испытания

1. **Прослеживаемость решений** — каждый диагностический вывод в Консилиуме сопровождается аудит-трейлом (какие специальности рассматривали кейс, какое было расхождение, эскалировался ли случай) — это ключевое требование к CDSS в клинических испытаниях.
2. **Встроенные защитные правила** — принудительная проверка врачом при низкой уверенности по тяжёлым диагнозам не зависит от согласия ИИ-агентов между собой.
3. **Многоуровневая анонимизация**, соответствующая ФЗ-152, с локальной (браузерной) обработкой части данных (RAG-библиотека, распознавание речи) — снижает объём персональных данных, покидающих контур врача.
4. **Формализованные клинические критерии** по каждой модальности (ЭКГ, лучевая диагностика, дерматоскопия, генетика) — не общий «спроси у ИИ», а структурированные, воспроизводимые протоколы анализа.
5. **Опциональные механизмы самопроверки** (Chain-of-Verification, RAG-gate) позволяют количественно оценивать частоту расхождений между первичным и перепроверочным ответом модели — готовая метрика для протокола испытания.
